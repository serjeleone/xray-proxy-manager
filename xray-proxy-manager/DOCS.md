# Xray Proxy Manager — техническая документация

## 1. Назначение

Xray Proxy Manager управляет Xray-конфигурациями, полученными из JSON-подписки, и предоставляет два независимых SOCKS5-слота. В каждый момент один слот считается активным, второй используется для подготовки следующей конфигурации.

Приложение не определяет способ подключения клиентов к слотам. Внешняя система может направлять новые соединения в активный слот любым подходящим способом. Для автоматического переключения приложение взаимодействует с совместимым selector API.

## 2. Архитектура

```text
JSON-подписка
  → разбор доступных outbound
  → проверка кандидатов
  → активный Xray-слот: xray-a или xray-b
  → резервный Xray-слот
  → внешний selector
```

Слоты используют отдельные Xray-процессы и отдельные конфигурационные файлы:

| Слот | Порт по умолчанию | Конфигурация |
|---|---:|---|
| `xray-a` | `10808` | `/config/xray-proxy-manager/config.xray-a.json` |
| `xray-b` | `10809` | `/config/xray-proxy-manager/config.xray-b.json` |

Состояние сохраняется в `/config/xray-proxy-manager`.

## 3. Переключение слотов

При выборе нового outbound приложение выполняет следующую последовательность:

1. Определяет резервный слот.
2. Формирует для него Xray-конфигурацию.
3. Выполняет `xray -test`.
4. Запускает резервный Xray-процесс.
5. Проверяет заданные URL через SOCKS5 резервного слота.
6. Переключает внешний selector на резервный слот.
7. Помечает прежний слот как завершающий соединения.
8. Контролирует соединения и счётчики трафика старого слота.
9. Останавливает старый процесс после устойчивого отсутствия активности.

Установленное соединение не переносится между Xray-процессами. Новые соединения после смены selector должны направляться в новый слот, а уже созданные могут завершиться через прежний.

Если проверка нового слота или обновление selector завершается ошибкой, активный слот не изменяется. В течение защитного периода после переключения приложение может вернуть selector на прежний работающий слот.

## 4. Требования к selector API

API задаётся параметром `selector_api_url` и должно быть доступно из контейнера приложения.

Используются следующие совместимые операции:

```text
GET /proxies/<selector_tag>
PUT /proxies/<selector_tag>
GET /connections
```

Ожидаемый ответ selector содержит поле `now`, равное `xray-a` или `xray-b`. Для выбора слота отправляется JSON:

```json
{
  "name": "xray-b"
}
```

Список соединений должен содержать массив `connections`. Для определения слота используется поле `chains` каждого соединения. Поля `upload` и `download` применяются для контроля изменения трафика.

Если endpoint `/connections` недоступен, приложение не останавливает старый слот автоматически.

## 5. Порты

| Порт по умолчанию | Протокол | Назначение |
|---:|---|---|
| `10808` | TCP/UDP | SOCKS5-слот `xray-a` |
| `10809` | TCP/UDP | SOCKS5-слот `xray-b` |
| `8099` | TCP, внутренний | Веб-интерфейс Home Assistant Ingress |

## 6. Установка через Home Assistant

Добавьте репозиторий:

```text
https://github.com/serjeleone/xray-proxy-manager
```

После установки укажите URL подписки и при необходимости параметры selector API. Внешний selector должен использовать точные теги `xray-a` и `xray-b` либо эквивалентную конфигурацию, возвращающую эти значения через API.

## 7. Параметры приложения

### Подписка и базовая конфигурация

| Параметр | Поддерживаемые значения | По умолчанию | Назначение |
|---|---|---|---|
| `subscription_url` | HTTP/HTTPS URL | пусто | JSON-подписка Xray. Обязательный параметр. |
| `config_index` | целое число от `0` | `0` | Предпочтительный индекс конфигурации при первом запуске. |
| `update_interval_hours` | `0…720` | `1` | Интервал обновления подписки. `0` отключает периодическое обновление. |
| `listen_lan` | `true`, `false` | `true` | `true` публикует SOCKS5-inbound на всех интерфейсах контейнера; `false` ограничивает loopback. |
| `socks_tcp_a` | TCP-порт | `10808` | Порт SOCKS5-inbound слота `xray-a`. Этот же номер используется UDP relay слота, если включён `socks_udp_a`. |
| `socks_tcp_b` | TCP-порт | `10809` | Порт SOCKS5-inbound слота `xray-b`. Должен отличаться от `socks_tcp_a`. Этот же номер используется UDP relay слота, если включён `socks_udp_b`. |
| `socks_udp_a` | `true`, `false` | `true` | Включает UDP relay в SOCKS5-inbound слота `xray-a`. |
| `socks_udp_b` | `true`, `false` | `true` | Включает UDP relay в SOCKS5-inbound слота `xray-b`. |

`socks_udp_a` и `socks_udp_b` не являются номерами портов. Каждый Xray-слот использует значение своего `socks_tcp_*` одновременно как TCP-порт SOCKS5 и как UDP endpoint для SOCKS5 UDP relay. Раздельные флаги позволяют независимо включать или отключать UDP в каждом процессе; для симметричной конфигурации рекомендуется оставлять оба значения одинаковыми.
| `override_inbounds` | `true`, `false` | `true` | Заменяет inbounds подписки управляемым SOCKS5-inbound. Для двухслотового режима должно быть `true`. |
| `proxy_username` | строка | пусто | Имя пользователя SOCKS5. Используется только вместе с `proxy_password`. |
| `proxy_password` | строка | пусто | Пароль SOCKS5. Используется только вместе с `proxy_username`. |

## 9. Восстановление

Если подписка временно недоступна, приложение использует сохранённую копию. Если новая конфигурация не проходит проверку, предыдущая рабочая конфигурация не заменяется.

При неизвестном состоянии selector приложение может временно запустить одинаковую конфигурацию в обоих слотах. После восстановления API фактический активный слот принимается как источник состояния, а дублирующий процесс завершается безопасно.

## 10. Ограничения

- Для бесшовного переключения требуется внешний selector, поддерживающий указанный API и учёт соединений.
- Приложение не переносит уже установленные соединения между процессами.
- При недоступном `/connections` старый слот сохраняется запущенным.
- UDP-состояния могут удерживать старый слот дольше TCP-соединений.
- Одновременное повторное переключение невозможно, пока резервный слот завершает старые соединения.
- При публикации SOCKS5-портов доступ следует ограничить доверенной сетью и при необходимости включить аутентификацию.
