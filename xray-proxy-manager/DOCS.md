# Xray Proxy Manager — техническая документация

## 1. Назначение

Xray Proxy Manager загружает Xray JSON-subscription, формирует конфигурации выбранных outbound и управляет двумя независимыми Xray-процессами. Один процесс обслуживает активный слот, второй используется для предварительной подготовки следующей конфигурации.

Приложение не определяет способ передачи клиентских соединений в SOCKS5-слоты. Для автоматического изменения активного слота используется совместимый selector API.

## 2. Архитектура

```text
JSON-subscription
  → разбор outbound
  → проверка кандидата
  → резервный Xray-слот
  → переключение selector
  → дренирование прежнего Xray-слота
```

| Слот | Порт по умолчанию | Конфигурация |
|---|---:|---|
| `xray-a` | `10808` | `/config/config.xray-a.json` |
| `xray-b` | `10809` | `/config/config.xray-b.json` |

Все постоянные данные находятся непосредственно в `/config`, который Home Assistant сопоставляет с каталогом конфигурации конкретного приложения. При первом запуске файлы из прежнего `/config/xray-proxy-manager` переносятся в `/config`, если целевые файлы ещё отсутствуют.

## 3. Переключение слотов

При выборе нового outbound выполняются следующие действия:

1. Проверяется, что outbound отличается от текущего активного.
2. Определяется свободный резервный слот.
3. При ручном выборе дренируемый резервный слот принудительно останавливается, если он мешает новой операции.
4. Создаётся отдельная конфигурация Xray для резервного слота.
5. Выполняются `xray -test` и сетевые проверки через SOCKS5 резервного слота.
6. Selector переводится на резервный слот.
7. Прежний активный слот помечается как дренируемый.
8. Новые соединения принимает новый слот; старый процесс продолжает обслуживать уже принятые соединения.
9. Старый слот останавливается после исчезновения отслеживаемых соединений и завершения защитного периода либо по достижении настроенного таймаута.

Открытое транспортное соединение не переносится между процессами. Переключение изменяет только назначение новых соединений.

Автоматический выбор не размещает один и тот же outbound одновременно в `xray-a` и `xray-b`. Если подходящей альтернативы нет, текущая конфигурация сохраняется до следующей проверки.

## 4. Дренирование

Основное условие штатной остановки старого слота:

```text
tracked_connections = 0
AND traffic_counters_are_stable
AND quiet_period_elapsed
```

Верхний блок и карточка соответствующего outbound показывают слот, число оставшихся соединений и кнопку принудительной остановки. Кнопка **«Стоп»** немедленно завершает Xray-процесс дренируемого слота вместе с оставшимися соединениями.

`drain_timeout_minutes` задаёт предельное время ожидания:

- `0` — неограниченное ожидание; соединения не завершаются приложением автоматически;
- `5` — короткий предел для тестовой среды;
- `15…30` — практичный диапазон для обычных клиентских соединений;
- `60` и более — консервативный вариант для длительных сессий.

Безопасное значение зависит от характера соединений. Для максимального сохранения длительных сессий используйте `0`.

## 5. Selector API

При `selector_control_enabled: true` приложение обращается к `selector_api_url` и использует `selector_api_secret` как Bearer-токен, если секрет задан.

Selector с именем `selector_tag` должен содержать:

```text
xray-a
xray-b
```

API должно обеспечивать:

- чтение текущего элемента selector;
- выбор `xray-a` или `xray-b`;
- получение списка соединений и цепочки outbound для определения дренирования.

Если selector API недоступно, новое двухслотовое переключение не начинается. Уже работающий Xray-процесс продолжает работу.

## 6. Опциональное управление правилом firewall для OpenWrt

Приложение может через SSH включать или отключать определенную секцию firewall на совместимом узле OpenWrt. Управление выполняется кнопкой play/pause и не изменяет настройки Xray, selector или подписки.

Поддерживаются методы `existing_key`, `generate_key` и `password`. Для режима `generate_key` приложение создаёт ключ в каталоге конфигурации и показывает открытую часть в интерфейсе. Открытый ключ требуется самостоятельно добавить в список разрешённых ключей удалённой системы.

Функция отключается параметром `router_control_enabled: false`.

## 7. Параметры конфигурации

| Параметр | Значения | По умолчанию | Назначение |
|---|---|---|---|
| `subscription_url` | URL-строка | пусто | Адрес JSON-subscription. Обязателен для обычного запуска. |
| `config_index` | целое число от `0` | `0` | Индекс конфигурации в корневом массиве подписки. |
| `update_interval_hours` | `0…720` | `1` | Период обновления подписки. `0` отключает расписание. |
| `override_inbounds` | `true`, `false` | `true` | Заменяет inbounds подписки двумя управляемыми SOCKS5-inbound. Для двухслотового режима должно быть `true`. |
| `disable_observatory` | `true`, `false` | `true` | Отключает observatory из загруженной конфигурации. |
| `log_level` | `debug`, `info`, `warning`, `error`, `none` | `warning` | Уровень журнала Xray. |
| `user_agent` | строка | `Xray Proxy Manager Home Assistant App` | User-Agent загрузчика подписки. |
| `validate_routing_tags` | `true`, `false` | `true` | Проверяет ссылки routing на существующие outbound. |
| `auto_fix_routing_tags` | `true`, `false` | `true` | Исправляет допустимые отсутствующие ссылки routing. |
| `auto_add_proxy_direct` | `true`, `false` | `true` | При необходимости добавляет служебный direct-outbound. |
| `restart_on_runtime_error` | `true`, `false` | `true` | Завершает контейнер при необрабатываемом падении активного Xray, если откат невозможен. |

## 8. Веб-интерфейс

Верхний блок содержит:

- кнопку play/pause для опционального удалённого управления правилом firewall на роутере OpenWrt;
- имя активного outbound;
- протокол, endpoint, outbound tag, активный слот и SOCKS5-порт;
- состояние дренируемого слота и число соединений;
- кнопки проверки, обновления подписки и принудительной остановки.

Карточки outbound показывают оранжевые индикаторы `A` и `B` для работающих слотов. Карточка дренируемого слота всегда получает соответствующую отметку и кнопку **«Стоп»**.

Окно логов поддерживает поиск, перенос строк, переход в начало и конец, копирование в буфер обмена и автоматическую загрузку текстового файла с датой в имени.

## 9. Хранение данных

| Путь | Назначение |
|---|---|
| `/config/subscription.json` | Последняя успешно загруженная подписка. |
| `/config/config.xray-a.json` | Конфигурация `xray-a`. |
| `/config/config.xray-b.json` | Конфигурация `xray-b`. |
| `/config/config.last_good.json` | Последняя рабочая конфигурация. |
| `/config/config.last_good.meta.json` | Метаданные последней рабочей конфигурации. |
| `/config/state.json` | Активный outbound, слот, задания и ошибки. |
| `/config/latencies.json` | Результаты проверок. |
| `/config/runtime-options.json` | Настройки, изменённые в интерфейсе. |
| `/config/ssh/` | Ключи для необязательного удалённого управления OpenWrt. |

Секреты и приватные ключи не выводятся в обычный статус API.

## 10. Восстановление и ограничения

- Если подписка недоступна, используется сохранённая копия.
- Неуспешная новая конфигурация не заменяет последнюю рабочую.
- При неизвестном состоянии selector запускается только запомненный активный слот; одинаковая конфигурация намеренно не запускается во втором слоте.
- При недоступном списке соединений автоматическая остановка дренируемого слота приостанавливается.
- UDP-состояния могут удерживать старый слот дольше TCP.
- Принудительная остановка или истечение `drain_timeout_minutes` может оборвать оставшиеся соединения.
- SOCKS5-порты следует публиковать только в доверенной сети либо защищать аутентификацией.

## 11. CI и релиз

`.github/workflows/validate.yml` проверяет Python, JavaScript, shell, YAML, JSON, полноту документации параметров и метаданные контейнера.

`.github/workflows/build.yml` собирает `amd64` и `aarch64` и публикует multi-arch образ:

```text
ghcr.io/serjeleone/xray-proxy-manager:v0.6.0
```
